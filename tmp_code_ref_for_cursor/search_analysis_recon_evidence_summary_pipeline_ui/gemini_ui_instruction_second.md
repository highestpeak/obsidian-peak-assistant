既然你的后端已经进化为一套**“侦察-归并-聚类-深度采证”**的精密流水线，UI 必须摆脱“对话框”的束缚，向**“任务指挥中心”**演进。

你可以参考以下这份 **UI 交互与视觉描述方案**，它可以直接作为给前端开发或设计工具（如 v0, Cursor, 或 Figma）的提示词。

---

### 1. 核心布局：双层结构 (Dashboard Style)

* **左侧/顶部（状态流）**：一个高度概括的阶段进度条（Classify -> Recon -> Planning -> Evidence）。
* **中心区域（动态看板）**：随着搜索进行，从“线索流”平滑演变为“任务卡片组”。

---

### 2. 关键阶段的 UI 表现描述

#### **阶段 A：维度解构 (Dimension Pulse)**

* **视觉**：用户输入问题后，中心区域出现一组彩色标签。
* **交互**：15 个维度（如“变现逻辑”、“风险评估”）像脉冲一样跳动。
* **状态文本**：`“正在解构问题语义，识别 12 个关键侦察维度...”`

#### **阶段 B：流式侦察 (Recon Lead Stream)**

* **视觉**：一个瀑布流列表，快速滚动显示被 AI 扫过的文件路径。
* **细节**：每个路径前面有一个小雷达图标。
* **交互**：用户点击路径，能看到一句话的 `tactical_summary`。
* **状态文本**：`“正在全库扫描... 发现 42 条潜在关联线索。”`

#### **阶段 C：智能归并与装箱 (Semantic Grouping)**

* **视觉（重点）**：原本散乱的“线索流”开始**向中心收缩**，聚合成 3-5 个巨大的卡片。
* **卡片内容**：
* **标题**：`topic_label`（例如：CMS 架构与备份安全）。
* **副标题**：`group_focus` 的核心短语。
* **清单**：列出该组包含的 3-5 个文件 Task（带状态勾选框）。


* **动画**：使用类似于“磁铁吸附”的效果，展示孤岛文件如何被吸入语义组。

#### **阶段 D：深度采证执行 (Evidence Mining)**

* **视觉**：当前正在处理的卡片高亮，其他卡片置灰。
* **细节**：卡片内部出现一个**微型终端窗口**，显示 `“Reading [file_A.md]... Extracting facts...”`。
* **结果实时弹出**：每提取到一个 `fact`，在卡片下方弹出一个小的气泡，点击气泡显示 `quote`（原文引用）。

---

### 3. 数据驱动的 UI 状态机 (给开发的 Prompt)

你可以用这段话描述交互逻辑：

> “UI 需要监听 `type` 为 `PLAN_CREATED` 的事件，一旦触发，界面必须从‘列表模式’切换为‘看板模式’。每个任务组卡片是一个独立的进度单元。
> 当收到 `EVIDENCE_DONE` 事件时，对应的 TaskID 应该在卡片清单中打勾。当整个卡片的所有 Task 完成，卡片边框变为绿色，并展示一个‘查看本组摘要’的按钮。”

---

### 4. 视觉风格建议 (Visual Style)

* **色调**：深色模式 (Space Gray)，科技感蓝/紫渐变作为强调色。
* **字体**：等宽字体（Monospace）用于显示路径和 TaskID，提升“工具感”。
* **组件压力**：在 Planning 阶段，使用 **骨架屏 (Skeleton Screens)** 预留卡片位置，减少布局跳动感。

---

### 5. 最终“合拢”动画

当最后一个 Evidence Pack 提交后：

1. **看板收起**：所有的任务卡片收缩到屏幕一侧。
2. **报告生成**：右侧展示最终的结构化报告，报告中的每一段都可以通过点击，高亮显示它来自哪个 `topic_label` 组。
