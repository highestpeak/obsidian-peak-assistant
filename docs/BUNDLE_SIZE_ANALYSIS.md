## Root causes

1. **sql.js WASM inlined**  
   The wasm-inline plugin in `esbuild.config.mjs` converts the full WASM to Base64 and inlines it into `SqlJsStore.ts`, ~860 KB.

2. **Streamdown CSS inlined**  
   `streamdown-isolated-css.ts` is generated by `scripts/concat-css.mjs` and embeds the full CSS as a string in the bundle (~1.45 MB).

3. **Multiple copies of lucide-react**  
   The main package, `@lobehub/ui`, and `@lobehub/icons` each bundle their own lucide-react, ~2.4 MB total.

4. **React DOM development build**  
   Without forcing production, the bundle includes `react-dom.development.js`, adding ~1 MB.

5. **Shiki full bundle**  
   `@shikijs/langs` (many languages) and `engine-oniguruma` are all included in the main bundle.

6. **Heavy libs not code-split**  
   cytoscape, mermaid, katex, handlebars, ai, langsmith, etc. are all pulled into the main entry.

---

## Optimization suggestions (by priority)

1. **Force React production build**  
   Add `define: { 'process.env.NODE_ENV': '"production"' }` in esbuild and ensure resolution picks `react-dom.production.min.js` (e.g. via `mainFields` / `conditions`).

2. **Dedupe lucide-react**  
   In esbuild or package.json, resolve `@lobehub/ui` and `@lobehub/icons` to the same lucide-react at project root (alias or overrides) so only one copy is bundled.

3. **Externalize Streamdown CSS**  
   Move the large CSS in `streamdown-isolated-css.ts` to a separate `.css` file and load it at runtime (e.g. via plugin asset path or Obsidian), saving ~1.45 MB in the main bundle.

4. **Externalize sql.js WASM**  
   Load the WASM at runtime from the plugin directory (or CDN) instead of inlining Base64 in the bundle, saving ~860 KB.

5. **Shiki on-demand / lazy load**  
   Bundle only the languages you use, or put Shiki behind a dynamic `import()` so it loads only when the syntax-highlight UI is used.

6. **Lazy load heavy libs**  
   Use dynamic `import()` for cytoscape, mermaid, katex, AI SDK, etc. on the screens that need them, and keep the main bundle to the entry and light logic.

7. **Inspect metafile regularly**  
   Production build already writes a metafile; run `BUNDLE_METAFILE=1 node esbuild.config.mjs production` to print the current size breakdown (see esbuild.config.mjs).

## Mermaid `window.load` listener leak (fixed)

Mermaid.js registers a `window` `load` listener when initialized with `startOnLoad: true`. Each plugin reload added another listener; those listeners held references to the old plugin context and caused large memory retention (~500MB per reload).

**Fix applied:**

- **StreamdownIsolated.tsx**: `mermaid.config.startOnLoad` set to `false` so Streamdown’s Mermaid plugin does not register a `window.load` listener.
- **mermaid-utils.ts**: Call `mermaid.initialize({ startOnLoad: false })` at module load so the first use of Mermaid (e.g. validation) never enables auto-run on load.
- **main.ts onunload**: Optional clear of `globalThis.lc_block_translators_registry` (LangChain) to avoid holding references.

`getEventListeners` is a DevTools-only API, so existing zombie listeners cannot be removed from plugin code. After this fix, new reloads no longer add listeners; a full Obsidian restart clears any that were added before the fix.
