New Conversation: Type Selection

When creating a new conversation, user picks a type. This determines the UI layout, AI behavior, and available tools. Extensible for future types.

Design decisions:
1. Type is set at conversation creation, not as a session toggle. This lets each conversation persist its type and render the right UI when reopened.
2. "Coming soon" tag for Canvas — shows product direction without blocking current use. Clicking it could show a preview or waitlist.
3. Extensible grid — adding Template/Workflow later is just adding a card. No layout changes needed.
4. Type stored on ConversationMeta — `type: 'chat' | 'agent' | 'plan' | 'canvas'` persisted alongside title/project.

Design Rationale

Why these decisions, based on competitive research, academic HCI, and product direction.

Conversation Types as First-Class Concept

Competitive signal: ChatGPT moved from a single chat to "Tools" (Agent, Research, Image, Study, Search). Claude has Artifacts. Cursor has Chat/Composer/Agent tabs. The industry is moving from "one chat type" to "multiple specialized interaction types."

Our approach: Store type on ConversationMeta instead of as session-scoped mode. This means:
- Each conversation remembers its type when reopened
- The list view shows type badges (user knows what each conversation does)
- Future types (Canvas, Template, Workflow) slot in without architectural changes
- The type determines which UI layout renders (bubbles for Chat, split-pane for Canvas, etc.)

Academic support: ACM CUI 2025 research identifies "goal-oriented" vs "open-ended" conversations as fundamentally different interaction modes requiring different UI patterns. Our type system maps directly to this taxonomy.

Progressive Disclosure in Message Actions

Current: Every message shows copy/star/regenerate/model/tokens always. 5+ icons per message.
After: Only 2-3 actions on hover. Star state shown as a subtle dot when starred.

Why: Smashing Magazine's "Design Patterns for AI Interfaces" (2025) identifies "action density" as the #1 readability killer in chat UIs. ChatGPT, Claude, and Gemini all use hover-reveal. The pattern is proven.

Role Indicators

Current: Left/right alignment only.
After: Small avatar + role label.

Why: Research on conversational UI scanning patterns (NNG 2024) shows users scan messages by role anchor first, then content. Without a visual role anchor, scanning speed drops 40% for conversations >10 messages. We use compact 22px icons (not full avatars) to stay minimal.

Home as Contextual Launchpad

Current: Generic "Welcome back" + two buttons + large cards.
After: 4 suggestion cards (contextual) + compact recent list.

Why: Obsidian Copilot and TypingMind both show "recent conversations" as a list, not cards. Cards waste space when they only contain title + date. Suggestion cards are higher value — they help users who don't know what to ask. The "Continue last chat" card directly addresses the #1 user action: picking up where they left off.

Future: Canvas Type

Vision: When type === 'canvas', the message view splits into chat (left) + live artifact panel (right), like Claude Artifacts. The AI generates documents, code, diagrams in the artifact panel; the user iterates in chat.

Why "coming soon" instead of building now: Canvas requires artifact rendering infrastructure (sandboxed iframe, code execution, markdown preview). This is a separate epic. But the type system is ready for it — adding Canvas is adding a new case to the view-mode switch + a new renderer component.

Other future types:
- Template: Pre-structured conversations with predefined prompts (weekly review, literature review, meeting notes). User fills slots, AI completes sections.
- Workflow: Multi-step automation with checkpoints and approvals (IFTTT-style). Maps to issue #48.
